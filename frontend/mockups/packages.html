Packages are the behavior layer — the Lua that runs on the hosted engine. They live in the hosted package store (GET/POST /api/v1/packages); an engine session (POST /api/v1/sessions, one live session per package, lease-fenced) scans its root once at start and composes one static graph. Changing the set or topology applies on the next session restart (stop → poll stopped → start) — there is no hot-reload, and the package source tree is read-only at runtime. What you manage here is which packages load and how they wire — not their source.

Add package

create a package in the hosted store — POST /api/v1/packages
Name · unique on create
Files · the package root, inline
composed_deps · optional
Stored in the hosted package store as {name, files[], composed_deps[]} — unique on create: a duplicate name is a 409, and updates or deletes are available via API (UI coming soon). A package is loaded once at session start and composed into the one static graph — applying it = stop the session, start a new one (POST /sessions/:id/stop → poll stopped → POST /sessions). Create validates structure only (name [A-Za-z0-9_-]+ · ≥1 file · ≤256 files · ≤1 MiB/file · ≤12 MiB total · must contain departments/*/main.lua or raisers/*.lua); conformance runs at session start — the session passes through validating and lands failed with the error on the session record if it doesn't conform. The source tree is read-only at runtime.
