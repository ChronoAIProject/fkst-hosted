Add the worker's supervise loop, credential-refresh client, status reporting, and OS-truth re-adopt (#173, increment 5 of #151) — dormant until activation. A spawned engine now gets a worker-side supervise loop mirroring the control-plane driver (500ms status tick → `StatusReport`, a 200ms mint servicer, stdout-401 detection, a stop signal → `Released`), MINUS journaling (which awaits the journal-crate relocation). `WorkerAgent::refresh_credential` POSTs a fence-echoed `CredentialRefreshRequest` to the controller's `/internal/v1/credential-refresh` and writes the 0600 token via `fkst_engine::write_token_file` — the worker mints fresh GitHub-App tokens ONLY over the RPC and never holds the App private key; the JIT servicer authenticates the engine's request against the controller's `.mint-nonce`, and the 55-min proactive + reactive (stdout-401) refreshes share the driver's escalating cooldown. A `credentials: None` (stale fence) self-fences the loop (no token written); `gone: true` fails the session loudly (the worker never silently 401s). `report_status` reports each transition (best-effort, fence-echoed). On restart `scan_and_adopt` re-adopts live engines, and an adopted engine's refresh servicer is parked behind a `FenceConfirmer` seam until the controller re-confirms a live fencing id, so it can never mint on a dead worker's stale nonce. The worker still links no `mongodb`/`fkst-control-plane`; no secret is logged.
