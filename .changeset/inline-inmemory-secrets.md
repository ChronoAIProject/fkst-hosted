Make secrets database-free (issue #138): remove the persistent, envelope-encrypted vault collection and its `/api/v1/vault/entries` CRUD, and supply secrets **inline at goal trigger** instead. Secrets (and non-secret env variables) now ride the trigger request body as a `secrets` array (`{key, value, kind}`, `kind` defaulting to `secret`), are held by the controller **in memory only** (zeroizing `SecretString`s keyed by `(owner, repo scope)`), injected into the engine env at session spawn through the unchanged `list_for_scope` contract, and dropped from memory when the session terminates. `VaultService` becomes an in-memory store (`set_inline`/`clear_inline`/`list_for_scope`); the at-rest AES-256-GCM crypto, the Mongo repo, the `fkst:vault:*` permissions, and the `FKST_HOSTED_VAULT_MASTER_KEY` config are gone (the `FKST_HOSTED_VAULT_*` value/entry caps remain). The DB-free failover re-mint gap is closed without a datastore: the controller holds the triggering user's access token in memory and re-supplies it to a same-process failover so the per-session NyxID key can be re-minted — so a secret session survives a **worker** loss but **not** a controller restart (a documented limitation), where a session that previously minted a key escalates to Failed rather than running with broken auth.
