The committed GitHub progress-record file is now the SOLE machine-truth for a logical run (issue #139): the two Mongo journaling collections (`session_progress`, `run_journals`) and their indexes are removed, and the run-head pointers (`issue_number`, `last_comment_id`, `last_commit_sha`, `completed_count`) move into `.fkst-hosted/journal/<run_key>.json`. The `Journaler` is now store-free: an in-RAM `completed[]` skip-set (seeded from the committed file at bootstrap plus the controller's single-writer-per-run guarantee, #135) replaces the unique partial index for cross-session dedupe; the in-RAM buffer is the only pre-flush store, so a crash before flush re-executes that work on redo (engine git-idempotency). The CAS-merge / strict-`<` fencing / debounced-flush semantics are preserved exactly, and the additive run-head pointer fields keep the record schema at `@1` (a pre-pointer file deserializes with defaults). Bootstrap now retries the committed-file read after a 404 (`FKST_JOURNAL_BOOTSTRAP_READ_RETRIES`, default 3) for GitHub eventual consistency, and a rolling activity comment is upserted at most once per committed flush when the run has a known issue number (`FKST_JOURNAL_ACTIVITY_COMMENT_ENABLED`, default true) — it carries only counts / latest transitions, never raw event payloads, and never leaks the GitHub token.
