Controller worker reassignment + re-dispatch (#140b). When a worker dies (misses its heartbeat liveness TTL) or releases a session during a graceful drain, the controller now reassigns that work onto a surviving Active worker AND actually re-dispatches it: `ClaimMap::reassign` bumps the run fence, then the new async `SecretRedispatch` seam (real impl `DispatchRedispatch`) re-resolves a fresh `ResolvedDispatch` stamped with the bumped fence and queues it to the new worker's outbound control channel (delivered on its next heartbeat). The abrupt-death path runs from the stale-worker sweeper (`on_worker_dead` reassigns ALL of the dead worker's in-flight sessions); the graceful-drain path is per-session (`on_session_released` reassigns exactly the session a `Released` ack named, ~zero double-run window). Both placement and reassignment now skip Draining workers (the Active filter in `snapshot_loads` + `live_worker_loads`), so a draining worker stops receiving new and redo work immediately. A failed re-dispatch resolution leaves the claim reassigned-but-unrun (retriable on the next tick) rather than unwinding. All of this is gated behind `FKST_DISPATCH_MODE` (default off): with the flag off the reassignment driver is never constructed, the drain handlers log only, and the sweeper stays log-only — byte-identical to the prior behaviour. The fence-bump + the engine's git-idempotency keep a redo safe even in the brief abrupt-death double-compute window. No secret (token, dispatch payload, env profile) is ever logged. Controller stays database-free.
