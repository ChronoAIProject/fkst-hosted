Trust the NyxID proxy for authentication and enforce `fkst:*` RBAC permission strings, mirroring the Ornn downstream pattern (issue #113). fkst-hosted no longer authenticates users itself: it removes all JWKS fetching and user-token signature verification (deleting `auth/jwks.rs` and the JWT crypto in `auth/verify.rs`) and instead **decodes** the NyxID-proxy-injected `X-NyxID-Identity-Token` (with an `X-NyxID-User-Id` headers-mode fallback) тАФ safe because the backend has no public ingress and is reachable only via the in-cluster proxy, which already verified the caller. Authorization is now two-layered: an **action layer** that requires the matching `fkst:*` permission (e.g. `fkst:goal:trigger`, `fkst:session:stop`, `fkst:github:write`, `fkst:catalog:read`, `fkst:vault:delete`, `fkst:admin`) consumed from NyxID's `permissions[]` тАФ fkst-hosted owns the vocabulary, NyxID owns the assignment, with no local roleтЖТpermission matrix тАФ followed by the unchanged **object layer** (ownership / org role / fail-soft org lookups). `AuthContext` is reshaped (`email`/`display_name`, live `roles`/`permissions`, `user_access_token: Option<SecretString>` for NyxID/Ornn forwarding only); the old `fkst:admin` *scope* is replaced by the `fkst:admin` *permission*. Operators must assign the matching `fkst:*` permissions to NyxID org roles (Admin тЖТ all, Member тЖТ read+write+trigger, Viewer тЖТ read) to preserve current behaviour.
