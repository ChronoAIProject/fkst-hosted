Inject user-pinned Ornn skills/skillsets into the per-session codex and add a list-available-skills catalog API (issue #114). At session create / goal trigger a user may pin Ornn skills/skillsets by concrete `name@<major.minor>` version; the session driver fetches each as that user — via the per-session NyxID token (#111) through the NyxID proxy slug `ornn-api` — and installs it into the per-session `$CODEX_HOME` (#112) before the engine spawns, so the run's `codex` discovers them. Fetching the full verbatim package is a two-hop flow: hop 1 (proxied) `GET /skills/<name>?version=` returns a pre-signed package URL; hop 2 is a DIRECT, unauthenticated GET of that time-limited chrono-storage URL (the URL is treated as sensitive and never logged). Skillsets are expanded via their closure (members installed two-hop; the skillset's master prompt appended idempotently to `$CODEX_HOME/AGENTS.md` inside an `ornn-skillset:<name>` marker block, replaced on re-pin). Exec bits are preserved from the zip's recorded unix mode, else granted to `scripts/*` and shebang files; every entry path is path-traversal-guarded. A cross-selection version conflict (one skill name pinned at two versions) hard-fails, and any Ornn `404`/`403` aborts the session start loudly — a pinned capability is never silently dropped. The resolved (non-secret) pins are persisted on the session so a failover rebuild re-injects the identical set. New `GET /api/v1/catalog/{skills,skillsets}` (with `scope`/`system`/`kind`/`tags`/`q`/`page`) and `GET /api/v1/catalog/{skills,skillsets}/:name/versions` forward the caller's NyxID token to Ornn's search/versions per scope and pass Ornn's result (incl. 4xx/5xx) through verbatim — fkst-hosted does zero permission logic; Ornn enforces all visibility. No fkst-substrate engine change.
